ImgProcessor开发记录(tmp)

关于IO流的小坑记录

​ 这里打算记录一下：涉及到ImgProcessor的一个私有类ImgProcessError，希望可以记录下ImgProcessError的内容，实现向任意文件而不是控制台输出内容，这样的话，就必然要求我们的函数参数需要有std::ostream&或者是std::ostream*，这里，为了避免函数总是传递相同的参数，我这里打算使用指针来完成化简函数参数栈和缓存的作用，而不是转瞬即逝的引用

// My first record
void ImgProcessorError::OutPutError(std::ostream& osf, const char* msg) // 去除累赘的抽象类型
{
	osf << msg;
}

​ 但是发现，我总是需要给所有抽象层的函数都要传递一次引用，在图像操作里带来了大量的栈消耗，这里，我打算改用指针，设计类的getter和setter接口缓存我们的IO流

void ImgProcessorError::BindErrorLog(std::ostream* osf) // 去除累赘的抽象类型
{
	this->os = osf; // a member called os, whose type is std::std::ostream*;
}

​ 这里就出现了一个这样的问题，那就是我们难以验证std::ostream的有效性：

#include<iostream>
#include<fstream>
const char* path = "D:\\Server_Related\\errorLog.txt";
struct T{
    std::ofstream* os;
};


void makeTmp(T* pt){
    std::ofstream ofs;
    ofs.open(path);
    pt->os = &ofs;
    if(ofs.is_open())
        std::cout << "ofs_open\n";
    else
        std::cout << "ofs_open failed\n";
}

int main()
{
    T t;
    makeTmp(&t);
    if((t.os)->is_open())
        std::cout << "ofs_open";
    else
        std::cout << "ofs_open failed";
}

​ 我们来看一段这样的代码——发现问题了嘛？是的，很经典，那就是绑定了一个转瞬即逝的参数给了一个生命周期更久的T的实例t,访问之，就会发现惊喜的炸了。

​ 如何检测呢，就是利用上面这个性质即可。

struct T{
    std::ofstream* os;
    bool isOSAvailable(){
    	return os->is_open();
    }
};
